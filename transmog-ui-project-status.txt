2. WHAT IS IMPLEMENTED BUT NEEDS WORK (Fragile/Incomplete)
================================================================================

A) PACKET PARSING — HEURISTIC / REVERSE-ENGINEERED
   File: TransmogrificationPackets.cpp
   ---------------------------------------------------------------
   The Read() methods for CMSG_TRANSMOG_OUTFIT_NEW and UPDATE_INFO use
   heuristic parsing rather than clean field-by-field reads:

   - TransmogOutfitNew::Read():
     Reads NPC guid, then takes a raw span of remaining bytes.
     Scans BACKWARDS from end for printable ASCII to find outfit name.
     Hardcodes middle section as exactly 6 bytes: [type:u8][flags:u8][icon:u32].
     Sets ParseSuccess=false on mismatch instead of throwing.

   - TransmogOutfitUpdateInfo::Read():
     Same pattern. Middle section is 5 bytes: [type:u8][icon:u32] (no flags).

   - TransmogOutfitUpdateSlots::Read():
     More structured: SetID, slotCount, Npc guid, then 16 bytes per slot.
     Has alignment-gap handling for unknown extra bytes between guid and slots.
     Each slot: AppearanceID(u32), RawSlotField(u32), Reserved1(u32), Reserved2(u32).
     Slot index extracted via RawSlotField >> 24.

   - TransmogOutfitUpdateSituations::Read():
     Clean structured read. SetID, Npc, count, then 16-byte entries.
     This is the only parser with no heuristic logic.

   All parsers store diagnostic fields (ParseError, DiagnosticReadTrace,
   PayloadSize, PayloadPreviewHex) for debugging.

   RISK: The hardcoded byte sizes for middle sections in NEW/UPDATE_INFO
   may break if the client's wire format changes in a patch.

B) SLOT MAPPING BUG — TransmogOutfitSlotToEquipSlot()
   File: TransmogrificationPackets.cpp
   ---------------------------------------------------------------
   The mapping function maps transmog outfit slots (0-14) to equipment slots.
   BUG: Both transmog slot 1 AND slot 2 map to EQUIPMENT_SLOT_SHOULDERS.

   The retail TransmogOutfitSlot enum has:
     0 = Head, 1 = ShoulderRight, 2 = ShoulderLeft, 3 = Chest, ...

   Slot 1 (ShoulderRight) → EQUIPMENT_SLOT_SHOULDERS is correct.
   Slot 2 (ShoulderLeft) should map to the secondary shoulder appearance
   (not the same equipment slot), or be handled specially.

   Currently slot 2 silently overwrites slot 1's shoulder value.

C) SMSG RESPONSE PACKETS — MINIMAL
   File: TransmogrificationPackets.cpp (lines 388-414)
   ---------------------------------------------------------------
   All four SMSG_TRANSMOG_OUTFIT_* Write() methods serialize only:
     uint32 SetID + uint64 Guid

   This may be sufficient if the client only needs an ACK, but retail
   responses may include additional data. Needs packet capture verification.

   Opcodes:
   - SMSG_TRANSMOG_OUTFIT_NEW_ENTRY_ADDED    = 0x42004A
   - SMSG_TRANSMOG_OUTFIT_INFO_UPDATED       = 0x42004B
   - SMSG_TRANSMOG_OUTFIT_SITUATIONS_UPDATED = 0x42004C
   - SMSG_TRANSMOG_OUTFIT_SLOTS_UPDATED      = 0x42004D

A) SITUATION PERSISTENCE
   File: TransmogrificationHandler.cpp, HandleTransmogOutfitUpdateSituations()
   ---------------------------------------------------------------
   The handler parses situation entries and sends an ACK, but the data
   is discarded. Explicit comment: "(stored as no-op; TODO implement
   situation persistence)"

   Situations are the retail auto-outfit-switching system. Each situation
   links a TransmogSituation enum value to a spec/loadout/equipment set:

   TransmogSituation enum (0-21):
     0=None, 1=Spec1, 2=Spec2, 3=Spec3, 4=Spec4,
     5=InCapitalCity, 6=InBattlefield, 7=OnDragonback, 8=InRaid,
     9=InDungeon, 10=Swimming, 11=Mounted, 12=InCombat,
     13=BearForm, 14=CatForm, 15=FlightForm, 16=MoonkinForm,
     17=TreeOfLifeForm, 18=GhostWolfForm, 19=TravelForm,
     20=StagForm, 21=MetamorphosisForm

   Needs:
   - DB table (character DB) to persist situation entries per outfit
   - Load at login, sync to client
   - Runtime auto-switch logic when situation changes (zone, spec, form, etc.)

   Priority: LOW for roleplay server. Players can manually equip outfits.

   Test: Create an outfit via the transmog UI, then delete it. Verify
   the client sends CMSG_DELETE_EQUIPMENT_SET and the server removes it.

5. KEY ARCHITECTURE NOTES
================================================================================

- Transmog outfits are stored as EquipmentSetInfo with Type=TRANSMOG
  (vs Type=EQUIPMENT for normal equipment sets). They share the same
  DB table (character_equipmentsets) and load/save pipeline.

- The client uses a unified C_TransmogOutfitInfo API for all outfit
  operations. The CMSG opcode determines what's being updated.

- SetID is server-assigned via FindNextAvailableTransmogSetID().
  The Guid field in SMSG responses is the player's uint64 GUID.

- Login delivery: outfits load via _LoadTransmogOutfits() and sync
  to the client via SMSG_LOAD_EQUIPMENT_SET (same as regular sets).

- The spell effect EQUIP_TRANSMOG_OUTFIT (347) is triggered when
  the player clicks "Apply" — the client casts spell 1247613 with
  the outfit's SetID as MiscValue[0].

- Appearance validation at save time uses sItemModifiedAppearanceStore
  and CollectionMgr::HasItemAppearance(). At equip time (spell effect),
  validation is intentionally skipped (assumed valid from save).
